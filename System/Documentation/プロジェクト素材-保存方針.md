# プロジェクト素材-保存方針

## ✅ 結論（おすすめ）

- **SecondBrain（Vault）**: 管理/思考の情報（Markdown中心: `Atlas/`・`Calendar/`・`Efforts/`）
- **Workspaces（ローカル）**: 実行するプロジェクト（コード/依存/サンプル/生成物）
- **Finals(OneDrive)**: 完成した成果物の正本
- Vault側は「リンク・参照情報」で繋ぐ（実体を入れない）

---

## 🔗 接続ポイント（Vault側）

各プロジェクトの `Efforts/<project>/00-概要.md` に、外部リソースを固定で書きます。

例:
- Repo（コード）: `~/Workspaces/projects/<project>/`
- GitHub URL:
- Samples（入力/fixture）: `~/Workspaces/projects/<project>/samples/`
- Deliverables（成果物出力）: `~/Workspaces/projects/<project>/deliverables/`
- Finals(OneDrive):（正本の保存先）
- 共有URL（Drive / Notion / Figma 等）:

---

## 📌 Workspaces（ローカル）に置くもの（推奨）

- コードリポジトリ（`node_modules/` / `.venv/` / `dist/` などが発生するもの）
- 大量・高頻度で更新されるファイル（ログ、キャッシュ、ビルド成果物、DB、データセット）
- 重い素材（動画、音声、RAW画像、Figma/PSD/AI等）
- 実データ（CSV/Excel）
  - Git混入を避けるため、`samples/` と `deliverables/` は **Git除外** を基本にする

---

## 📌 Vaultに置いて良いもの（推奨）

- 企画・要件・議事録・意思決定ログ・チェックリスト（`.md`）
- すぐ参照したい小さめ添付（スクショ等）→ `Calendar/inbox/`（Obsidianの添付保存先）
- 手順書/README（参照版）→ `Atlas/`（正本が `Finals(OneDrive)` の場合は注記してリンクのみ）

---

## 🤖 AIツール運用の観点

- **Claude Code / Codex** は基本的にローカルRepo（`~/Workspaces/projects/<project>/`）で回す
- Vault側は「目的/制約/意思決定ログ/参照」を置く
- ツール切替の再開SSOTは **repo直下の `handoff.md`**（`handoff` / `endwork`）
